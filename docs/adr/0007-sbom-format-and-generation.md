# ADR-0007: Generate CycloneDX SBOMs From the Final Pushed Image, via Syft

**Status:** Accepted
**Date:** 2026-07-12

## Context

An SBOM's value is in enabling fast, later queries ("does any shipped image
contain this vulnerable library version") without rebuilding, and in serving
as the shared input document for downstream scanning (Grype) and attestation
(Cosign). Two standard formats compete: SPDX (Linux Foundation origin,
stronger on licensing/compliance) and CycloneDX (OWASP origin, richer security
and vulnerability metadata, the format most security tooling targets first).

## Decision

Generate the SBOM with Syft in **CycloneDX JSON** format, from the final built
and pushed image referenced **by digest**, not from source and not from a tag.

## Consequences

- Aligns with the security-tooling-first framing of this project — CycloneDX
  is the format most vulnerability-scanning and attestation tooling (including
  Grype) is built around.
- Scanning the pushed image by digest, rather than source, ensures the SBOM
  reflects what actually ships (base image OS packages, anything added during
  build) rather than only what was written in the repo.
- Syft's static inspection (package-manager metadata, no code execution) makes
  it safe to run against an already-built, potentially untrusted artifact.
- SPDX is not generated by default under this decision; if a future compliance
  requirement needs it specifically (e.g. NTIA-minimum-elements conformance),
  Syft can emit it from the same scan without additional tooling.

## Alternatives Considered

- SPDX as primary — viable, stronger compliance/licensing framing; not chosen
  because this project's stated audience is security/DevOps interviewers, for
  whom CycloneDX is the more immediately recognizable choice.
- Generating from source pre-build — rejected: does not reflect the shipped
  artifact's actual contents.
