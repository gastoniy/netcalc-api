# ADR-0009: SLSA Provenance via a Separate, Minimally-Privileged Step — and an Enforced Verification Gate

**Status:** Accepted
**Date:** 2026-07-12

## Context

SLSA's core trust argument at higher build levels requires that provenance be
generated by something isolated from the build job itself — if the same job
that compiles the code also self-asserts its own legitimacy, a compromised
dependency inside that build could forge the provenance too. Separately,
producing a signature and provenance document is worthless as a security
control unless something actually checks them and rejects on failure — signing
without a consuming verification step is decoration, not a gate.

## Decision

Generate SLSA provenance using GitHub's native `actions/attest-build-
provenance` action, pointed at the pushed image's digest, rather than hand-
rolling the Fulcio/Rekor calls. Add an explicit verification step
(`cosign verify` + `cosign verify-attestation`) as its own pipeline stage,
run against the specific expected `--certificate-identity` and
`--certificate-oidc-issuer`, with a non-zero exit code failing the job.

## Consequences

- `actions/attest-build-provenance` runs as a distinct action invocation
  receiving only the digest, not arbitrary build-job permissions — satisfying
  the isolation principle without hand-building a custom reusable workflow.
- The verification step is the actual enforced control in this pipeline;
  without it, signing and provenance generation are audit trail only, not
  policy enforcement. This is the single most commonly-skipped step in junior
  implementations of this pattern.
- For a real deployment target (not just CI), the same verification logic
  belongs as a Kubernetes admission policy (e.g. Kyverno or Sigstore Policy
  Controller) so unsigned or unverifiable images are rejected at deploy time,
  not just flagged in a build log.

## Alternatives Considered

- `slsa-framework/slsa-github-generator` reusable workflow — more explicit
  about the isolated-builder model; not chosen as primary for simplicity, but
  worth demonstrating as an alternative to show deeper SLSA Build L3
  understanding if time allows.
- Signing/attesting without a verification step — explicitly rejected; this is
  the failure mode this ADR exists to prevent.
